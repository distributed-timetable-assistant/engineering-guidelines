راهنمای مهندسی DiTA

زبان‌های برنامه‌نویسی

به بخش زبان‌های دستورالعمل‌های مهندسی Dita خوش آمدید. در اینجا استانداردها، بهترین روش‌ها و الزامات ابزار برای زبان‌های برنامه‌نویسی که استفاده می‌کنیم را خواهید یافت.

زبان‌های پشتیبانی شده

  • Rust - دستورالعمل‌هایی برای نوشتن سیستم‌های ایمن، همزمان و کاربردی در Rust.

اهداف این بخش:

  • تضمین یکپارچگی در تمام کدبیس‌ها.
  • ترویج روش‌های اصولی مختص هر زبان.
  • تعریف ابزارها و پیکربندی استاندارد (لینترها، فرمت‌کننده‌ها).

دستورالعمل‌های مهندسی Rust

این بخش استانداردها و بهترین روش‌ها را برای توسعه Rust در Dita شرح می‌دهد. هدف ما بهره‌گیری از تضمین‌های ایمنی Rust در عین حال حفظ کدی خوانا و قابل نگهداری است.

اصول اصلی

  1. ایمنی در اولویت (Safety First): از کد unsafe اجتناب کنید مگر اینکه کاملاً ضروری باشد و به شدت ایزوله و مستند شده باشد.
  2. راست اصولی (Idiomatic Rust): از قراردادهای استاندارد پیروی کنید. اگر در حال «جنگیدن با borrow checker» هستید، طراحی را بازنگری کنید.
  3. مستندسازی (Documentation): تمام APIهای عمومی باید مستند شوند.

ابزارها

  • فرمت‌کننده (Formatter): ما از rustfmt با پیکربندی استاندارد استفاده می‌کنیم.
  • لینتر (Linter): ما از clippy استفاده می‌کنیم. پایپ‌لاین‌های CI باید در صورت وجود هشدارهای clippy با خطا مواجه شوند.

موضوعات

  • ساختار پروژه (اضافه خواهد شد)
  • مدیریت خطا (اضافه خواهد شد)
  • تست (اضافه خواهد شد)
  • Async/Await (اضافه خواهد شد)

سوابق تصمیم‌گیری معماری (ADRs)

سوابق تصمیم‌گیری معماری (ADRs) اسناد متنی کوتاهی هستند که یک تصمیم معماری مهم گرفته شده را به همراه زمینه و پیامدهای آن ثبت می‌کنند. ما از ADRها برای پیگیری تاریخچه تصمیماتمان و استدلال‌های پشت آن‌ها استفاده می‌کنیم.

مکان فایل

تمام ADRها باید در پروژه doc قرار گیرند.

قرارداد نام‌گذاری

فایل‌های ADR باید با استفاده از یک شماره ترتیبی و یک عنوان کوتاه توصیفی، که با خط تیره جدا شده‌اند، نام‌گذاری شوند. فرمت به این صورت است: NNNNNN-title-of-the-decision.md.

مثال: 000001-record-architecture-decisions.md

وضعیت‌ها

یک ADR می‌تواند یکی از وضعیت‌های زیر را داشته باشد:

  • پیشنهاد شده (Proposed): تصمیم در حال بحث است و هنوز توافق نشده است.
  • پذیرفته شده (Accepted): تصمیم توافق شده و باید پیاده‌سازی شود.
  • رد شده (Rejected): تصمیم پیشنهاد و بحث شده اما پذیرفته نشده است. سابقه برای زمینه تاریخی نگهداری می‌شود.
  • منسوخ شده (Deprecated): تصمیم در گذشته پذیرفته شده اما دیگر قابل اجرا یا توصیه شده نیس.
  • جایگزین شده (Superseded): تصمیم با یک تصمیم جدیدتر جایگزین شده است. ADR جدید باید به ADR جایگزین شده ارجاع دهد.

قالب (Template)

از قالب زیر برای ADRهای جدید استفاده کنید:

# NNNN. Title of the Decision

Date: YYYY-MM-DD

## Status

[Proposed | Accepted | Rejected | Deprecated | Superseded]

## Context

What is the issue that we're seeing that is motivating this decision or change?

## Decision

What is the change that we're proposing and/or doing?

## Consequences

What becomes easier or more difficult to do and any risks introduced by the change that will need to be mitigated.

نحوه ارسال

  1. یک شاخه جدید برای ADR خود بسازید.
  2. فایل ADR جدید را در پروژه doc اضافه کنید.
  3. اطمینان حاصل کنید که نام فایل از شماره در دسترس بعدی استفاده می‌کند.
  4. یک Pull Request برای بازبینی ارسال کنید.

جریان کاری گیت

این بخش جریان کاری گیت ما را که شامل شاخه‌گذاری، کامیت کردن و مدیریت موضوعات (issues) است، مشخص می‌کند.

  1. شاخه‌گذاری: برای هر وظیفه یک شاخه جدید بسازید. به استراتژی شاخه‌گذاری مراجعه کنید.
  2. کامیت کردن: پیام‌های کامیت واضح و طبق قرارداد بنویسید. به راهنمای کامیت مراجعه کنید.
  3. ایخوها (Issues): کار خود را به ایشوهای مشخص لینک کنید. به مدیریت ایشوها مراجعه کنید.
  4. ADRها: در صورت لزوم به تصمیمات معماری ارجاع دهید. به لینک‌دهی ADR مراجعه کنید.

مراحل جریان کاری

  1. یک ایشو انتخاب کنید (یا یکی بسازید).
  2. یک شاخه از dev (برای ویژگی‌های جدید) یا master (برای رفع حوادث) بسازید.
  3. تغییرات را پیاده‌سازی کنید.
  4. تغییرات را با پیروی از راهنمای کامیت کامیت کنید.
  5. شاخه را پوش کنید و یک Pull Request باز کنید.
  6. اطمینان حاصل کنید که بررسی‌های CI (شامل commitlint) پاس شوند.

استراتژی شاخه‌گذاری

ما از استراتژی استفاده می‌کنیم که امکان توسعه موازی با استفاده از شاخه‌های کوتاه-مدت را فراهم می‌کند. مخزن دارای دو شاخه دائمی است: master و dev.

شاخه‌های اصلی

  • master: وضعیت آماده برای تولید. شامل کد پایدار است.
  • dev: شاخه اصلی توسعه. تمام ویژگی‌های جدید ابتدا در اینجا ادغام می‌شوند.

نام‌گذاری شاخه‌ها و استراتژی

شاخه‌های ویژگی (Feature Branches)

برای اضافه کردن قابلیت‌های جدید استفاده می‌شود.

  • شاخه منبع: dev
  • قرارداد نام‌گذاری: باید شامل شماره ایشو باشد.
  • فرمت: feat/issue-<number>/<short-description>
  • مثال: feat/issue-42/add-login-page

شاخه‌های رفع باگ (Bug Fix Branches)

برای رفع باگ‌های حیاتی یا حوادث در محیط تولید استفاده می‌شود.

  • شاخه منبع: master (برای هات‌فیکس‌ها/حوادث ترجیح داده می‌شود)
  • قرارداد نام‌گذاری: باید شامل شماره حادثه باشد.
  • فرمت: fix/incident-<number>/<short-description>
  • مثال: fix/incident-808/resolve-memory-leak

سایر شاخه‌ها

برای مستندات، ریفکتورینگ یا کارهای روزمره (chores).

  • شاخه منبع: dev (معمولاً)
  • فرمت: type/<scope>/<short-description> یا type/issue-<number>/<short-description>
  • مثال: docs/readme/update-setup

شاخه‌ها باید پس از ادغام حذف شوند.

راهنمای کامیت

پیام‌های کامیت باکیفیت برای حفظ یک کدبیس سالم ضروری هستند. آن‌ها به توسعه‌دهندگان جدید کمک می‌کنند تا زمینه را درک کنند، دیباگ کردن را با git blame ساده می‌کنند و امکان تولید خودکار لاگ تغییرات (changelog) را فراهم می‌کنند.

ساختار پیام

یک پیام کامیت از یک هدر، بدنه و پاورقی تشکیل شده است.

<type>(<scope>): <subject>

<body (اختیاری، اما توصیه می‌شود)>

<footer (اختیاری)>

۱. هدر (The Header)

هدر اجباری است و باید ۵۰ کاراکتر یا کمتر باشد.

نوع (Type)

باید یکی از موارد زیر باشد:

دامنه (Scope)

دامنه اختیاری است اما توصیه می‌شود. "مکان" تغییر کامیت را مشخص می‌کند.

  • مثال‌ها: auth, api, ui, database, deps.
  • فرمت: feat(auth): ...

موضوع (Subject)

موضوع شامل توصیف مختصر تغییر است.

  • حالت امری: "Add" نه "Added"، "Fix" نه "Fixed".
  • بدون نقطه: در انتها نقطه نگذارید.
  • حروف کوچک: حرف اول معمولاً کوچک است (Conventional Commits این را اجبار نمی‌کند، اما یکپارچگی کلید است. نکته: "Seven Rules" می‌گوید با حروف بزرگ شروع کنید، مثال‌های Conventional Commits اغلب با حروف کوچک هستند. ما از حروف کوچک پیروی می‌کنیم تا با تنظیمات پیش‌فرض commitlint مطابقت داشته باشد، مگر اینکه طور دیگری تنظیم شده باشد.) -> تصحیح خودکار: فایل قبلی می‌گفت "Capitalize". قانون "Seven Rules" می‌گوید بزرگ بنویسید. Conventional Commits خنثی است. بیایید به Capitalize (حرف اول بزرگ) وفادار بمانیم تا در لاگ‌های گیت حرفه‌ای‌تر به نظر برسد و با قانون قبلی مطابقت داشته باشد.

۲. بدنه (The Body)

بدنه اختیاری است اما برای تغییرات غیربدیهی قویاً توصیه می‌شود.

  • شکستن خط در ۷۲ کاراکتر: این خوانایی را در تمام محیط‌ها تضمین می‌کند.
  • انگیزه: توضیح دهید چرا این تغییر را انجام می‌دهید.
  • تضاد: رفتار جدید را با رفتار قبلی مقایسه کنید.
  • توضیح: توضیح دهید چه چیزی و چرا، نه چگونه (کد، چگونه را توضیح می‌دهد).

۳. پاورقی (The Footer)

پاورقی برای اطلاعات متا استفاده می‌شود.

تغییرات شکننده (Breaking Changes)

تمام تغییرات شکننده باید در پاورقی با توصیف تغییر، توجیه و یادداشت‌های مهاجرت ذکر شوند.

  • فرمت: با BREAKING CHANGE: <description> شروع کنید.
  • جایگزین: اضافه کردن ! بعد از type/scope در هدر (مثلاً feat(api)!: ...).

ارجاعات

  • Issues: Closes #123, Fixes #42.
  • ADRs: ADR: 004.

همکاری

  • Co-authored-by: Co-authored-by: Name <name@example.com>

مثال‌ها

ویژگی با دامنه

feat(auth): Add Google OAuth login support

Add support for Google OAuth 2.0 to allow users to sign up
and log in using their Google accounts. This simplifies the
onboarding process.

Closes #101

رفع باگ با تغییر شکننده

fix(api): Handle null values in user response

Previously, the API crashed when the user had no address.
Now, it returns a null address field instead.

BREAKING CHANGE: The `address` field in the user object
can now be null. Clients must update their code to handle
this case.

مستندات

docs(readme): Update installation instructions

بازگردانی (Revert)

revert: let us never speak of this again

This reverts commit 6b2a412.

مدیریت ایشوها

لینک کردن صحیح فعالیت‌های گیت به ایشوها، تاریخچه‌ای شفاف و وضعیت‌های خودکار گردش کار را تضمین می‌کند.

لینک کردن در کامیت‌ها

به ایشوها در پاورقی (footer) کامیت خود ارجاع دهید.

بستن ایشوها

برای بستن خودکار یک ایشو هنگامی که کامیت ادغام می‌شود:

Closes #123

یا برای چندین ایشو:

Closes #123, #245

ارجاع به ایشوها

برای ارجاع ساده به یک ایشو بدون بستن آن (مثلاً "همچنین ببینید"):

Refs #123

لینک کردن در Pull Requestها

در توضیحات Pull Request خود، می‌توانید از کلمات کلیدی برای لینک کردن ایشوها استفاده کنید.

  • "Closes #123"
  • "Fixes #123"
  • "Resolves #123"

این کار یک لینک در گیت‌هاب ایجاد می‌کند و هنگامی که PR در شاخه پیش‌فرض ادغام شود، ایشو را به طور خودکار می‌بندد.

لینک‌دهی ADR

سوابق تصمیم‌گیری معماری (ADR) "چرایی" تغییرات مهم را توضیح می‌دهند. بسیار مهم است که تغییرات کد را به تصمیماتی که آن‌ها را مجاز کرده‌اند، لینک کنید.

لینک کردن ADRها در کامیت‌ها

اگر یک کامیت یک ADR خاص را پیاده‌سازی می‌کند، مشابه ایشوها، در پاورقی پیام کامیت به آن ارجاع دهید.

فرمت:

ADR: <ADR-Number>

مثال:

feat(database): migrate to postgres

We are migrating to Postgres to support better transaction handling as decided.

Closes #45
ADR: 0012

لینک کردن ADRها در ایشوها/PRها

هنگام باز کردن یک ایشو یا PR که مربوط به یک تصمیم معماری است، لینکی به فایل ADR در توضیحات اضافه کنید.

مثال:

Implements ADR-0012: Use Postgres
